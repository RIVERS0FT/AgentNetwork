# AI Agent 仿真运行平台

> **版本**: 0.3.0 | **技术栈**: Python / FastAPI / React / Docker Compose

AI Agent 仿真运行平台是一个**企业级多 Agent 仿真与推演平台**。用户通过场景文件夹定义角色、拓扑关系和技能，平台自动分配 Docker 容器，驱动 LLM 决策，实现多角色博弈推演，并在可视化地图上实时呈现。

---

## 一、核心能力

| 能力 | 描述 |
|------|------|
| **场景定义** | 场景文件夹（3 个 JSON + skills.py），定义角色、拓扑、技能 |
| **Agent 管理** | Docker 容器池 (brain/claude-code/openclaw)，动态分配 + 扩容 |
| **LLM 决策** | 每个 Agent 独立 LLM 大脑 (DeepSeek)，根据角色和目标自主决策 |
| **Agent 通信** | 消息总线 HTTP 中转，关系矩阵 = 通信矩阵（有 edge 才能通信） |
| **容器化运行** | Docker Compose 编排，3 种 Agent 后端，容器池复用 |
| **可视化** | Campus Atlas 地图 + Agent 实时位置 + 关系拓扑 + 悬停面板 + 日志面板 |
| **日志系统** | 结构化 JSON Lines 持久化，WebSocket 实时推送，支持导出 |

## 四、三种 Agent 后端

| 后端 | 容器前缀 | 镜像 | LLM 调用方式 |
|------|---------|------|-------------|
| **brain** | ag-b | Dockerfile | `brain.py` → OpenAI 兼容 API (httpx) |
| **claude-code** | ag-c | Dockerfile.claude | `claude -p` CLI 子进程 |
| **openclaw** | ag-o | Dockerfile.openclaw | Anthropic SDK (tool-use) |

三种后端均使用 **DeepSeek API** (`deepseek-chat`)，通过 `ANTHROPIC_BASE_URL=https://api.deepseek.com/anthropic` 访问 DeepSeek 的 Anthropic 兼容端点。

## 五、容器池机制

### 固定命名

docker-compose 使用 YAML anchors 定义固定容器名：

| 类型 | 容器名 |
|------|--------|
| brain | ag-b1, ag-b2, ag-b3, ag-b4 |
| claude-code | ag-c1, ag-c2, ag-c3 |
| openclaw | ag-o1, ag-o2, ag-o3 |
| 消息总线 | bus |
| 调度中心 | srv |

### 分配策略 (`container_runtime.py`)

1. 运行场景时，遍历场景 Agent，调用 `assign_agent(agent_id, role, name, extra_meta)`
2. `_get_or_create_container(backend)` 从对应类型的运行中容器里找第一个未分配的
3. 池耗尽 → 自动 `docker run` 创建新容器（命名 ag-{prefix}10+）
4. 场景结束后 `runtime.reset()` 释放占用标记，容器复用

每个场景 Agent 独占一个容器（不共享）。

## 六、Agent 注册与通信

### 注册流程

```text
场景启动 (_launch_containers)
  │
  ├─ assign_agent() → 分配容器, 获得 ca.url = "http://ag-c1:8000"
  │
  ├─ POST {ca.url}/reset → 清除上一轮残留 (brain 记忆 + inbox)
  │
  ├─ POST bus:9000/register?agent_id=ceo&url=http://ag-c1:8000&name=首席执行官
  │     └─ bus 注册表: agent_registry["ceo"] = "http://ag-c1:8000"
  │                     agent_registry["首席执行官"] = "http://ag-c1:8000"
  │
  └─ 开始仿真轮次
```

### 通信链路 (全部通过消息总线)

```text
Agent A /act
  │
  ├─ comm.send("cfo", "CFO", "ceo", "消息内容")
  │     └─ POST bus:9000/relay  {from_id:"cfo", to:"ceo", content:"..."}
  │           ├─ bus 查 agent_registry["ceo"] → "http://ag-c1:8000"
  │           └─ POST http://ag-c1:8000/message → Agent ceo inbox
  │
  ├─ comm.broadcast("ceo", "CEO", "全员通知", allowed={cto,cfo,pm})
  │     └─ POST bus:9000/relay  {from_id:"ceo", to:"broadcast", allowed:[...]}
  │           ├─ bus 遍历 agent_registry，过滤 allowed + 排除自己
  │           └─ 逐个 POST /message 到目标容器
  │
  └─ 通信权限: 关系矩阵 edge 定义可通信对象
       CEO ↔ CTO, CFO, PM, SALES_LEAD (有 edge)
       CEO → DEV_LEAD (无 edge → 拒绝: "无通信权限")
```

## 七、场景定义 (文件夹格式)

### 文件结构

```
scenes/network_collaboration/
├── meta_and_roles.json       # 角色定义 + 全局规则
├── instances_and_skills.json # 容器实例 + 技能绑定
├── network_topology.json     # 通信拓扑 (关系矩阵)
└── skills.py                 # Python 技能实现 (可选)
```

### meta_and_roles.json

```json
{
  "scenario_metadata": {
    "title": "企业协作仿真",
    "global_rules": "仿真周期90天。每轮代表3天..."
  },
  "roles": {
    "CEO": {
      "name": "首席执行官",
      "model_backbone": "claudecode",
      "identity": "公司CEO，负责整体战略和重大决策。",
      "core_goal": "90天内公司整体利润增长20%，市场份额提升5%。",
      "primary_interaction_paradigm": "INTERNAL_COLLABORATION"
    }
  }
}
```

- `model_backbone`: `claudecode` → claude-code 容器, `openclaw` → openclaw 容器
- role key 小写后作为 `agent_id` (CEO → ceo)

### network_topology.json (通信矩阵)

```json
{
  "sub_networks": [{
    "edges": [
      {"source": "CEO", "target": "CTO", "paradigm": "COLLABORATION"}
    ]
  }]
}
```

**关系矩阵 = 通信矩阵**：只有 edge 连接的 Agent 才能互发消息。双向通信。

## 八、LLM 集成

所有 Agent 使用 **DeepSeek API** (`deepseek-chat`)。

### 配置 (docker-compose.yml)

```yaml
environment:
  ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}    # DeepSeek key
  LLM_MODEL: deepseek-chat
  ANTHROPIC_BASE_URL: https://api.deepseek.com/anthropic  # DeepSeek Anthropic 兼容端点
```

### 决策流程

```
/decide (POST)
  ├─ 注入场景身份 (effective_id, core_goal, hidden_secret, action_space)
  ├─ 注入通信矩阵 (comm_matrix)
  ├─ 构建 prompt (系统提示 + 身份 + 已知 Agent + 收件箱)
  ├─ LLM 调用 → 解析 JSON Action
  └─ 返回 { type: "send_message", target: "cfo", content: "..." }

/act (POST)
  ├─ 检查通信权限 (target 是否在 allowed_targets 中)
  ├─ comm.send() / comm.broadcast() → bus relay
  └─ 记录日志 (success/failed)
```

## 九、仿真运行流程

### Setup → Launch 两步

```text
1. POST /api/simulations/setup  { scene: "network_collaboration" }
     └─ _setup_scene()
          ├─ 解析场景文件夹 → SceneDefinition
          ├─ 创建 Agent 对象 → AgentRegistry.register()
          ├─ 力导向布局计算坐标
          └─ 返回 Agent 列表 + 关系图

2. POST /api/simulations/launch
     └─ _launch_containers()
          ├─ assign_agent() 为每个 Agent 分配容器
          ├─ 容器状态重置 (POST /reset 清除 brain 记忆 + inbox)
          ├─ 注册到消息总线
          ├─ 构建通信矩阵 (workflow edges → comm_matrix)
          ├─ 运行 MAX_ROUNDS 轮:
          │    ├─ 注入 context (agents, tasks, comm_matrix)
          │    ├─ runtime.run_round() → decide_all() + act_all()
          │    └─ 每轮结束后 Agent 状态置为 idle（停止前端动画）
          ├─ 停止条件: 超过 stalemate_threshold 轮无变化则提前结束
          └─ 返回结果 + 日志

注：仅支持场景文件夹格式（scenes/<name>/ 目录）。
```

## 十一、日志系统（v2 统一 schema）

`global.jsonl` 是唯一全局时间线。`communication.jsonl` 和 `behavior.jsonl` 是同结构的过滤视图。

### 统一 schema 字段

每条日志记录包含以下顶层字段：

| 字段 | 类型 | 含义 |
|------|------|------|
| `timestamp` | string | ISO 8601 毫秒时间戳 |
| `seq` | int | 会话内自增序号 |
| `session_id` | string | 会话标识，如 `minesweeper_20260611_191402_501302` |
| `level` | string | `INFO` \| `WARN` \| `ERROR` |
| `source` | string | 来源：`backend` \| `agent` \| `bus` \| `frontend` |
| `component` | string | 组件：`srv` \| `bus` \| `dashboard` \| agent_id |
| `category` | string | 分类（决定文件路由，见下） |
| `event` | string | 事件名，如 `agent_message`、`session_start` |
| `actor` | object | 发起方，`{"id": "cmdr_01"}` |
| `target` | object | 目标方，`{"id": "soldier_07"}` |
| `action` | object | 动作，`{"name": "send_message", "status": "success"}` |
| `message` | string | 短摘要，仅用于人类快速阅读（不承载完整数据） |
| `payload` | object | 业务数据：消息正文、推理内容、技能参数/结果等 |
| `network` | object | 通信层数据：IP、端口、协议、延迟、包大小、TCP flags |
| `trace` | object | 追踪信息：轮次、会话、channel_id、correlation_id |

### category 分类与文件路由

| category | 含义 | 写入文件 |
|----------|------|---------|
| `communication` | Agent 间通信报文 | global.jsonl + **communication.jsonl** |
| `agent_behavior` | Agent 决策/执行（decide, act） | global.jsonl + **behavior.jsonl** |
| `system` | 系统事件（启动/完成/注册/错误/事件触发） | global.jsonl |
| `frontend` | 前端交互日志（WebSocket 连接、按钮点击） | global.jsonl |
| `lifecycle` | 容器/会话生命周期 | global.jsonl |

### agent_message 示例

```json
{
  "timestamp": "2026-06-11T19:14:02.501", "seq": 42,
  "session_id": "minesweeper_20260611_191402_501302",
  "level": "INFO", "source": "bus", "component": "bus",
  "category": "communication", "event": "agent_message",
  "actor": {"id": "cmdr_01"},
  "target": {"id": "soldier_07"},
  "action": {"name": "relay", "status": "delivered(200)"},
  "message": "cmdr_01 → soldier_07",
  "payload": {
    "content": "报告指挥官，侦察任务已完成...",
    "reasoning": ""
  },
  "network": {
    "src_ip": "172.19.0.4", "src_port": 0,
    "dst_ip": "172.19.0.8", "dst_port": 0,
    "protocol": "TCP/HTTP", "latency_ms": 8.4,
    "packet_len": 317, "header_len": 200, "payload_len": 117,
    "tcp_flags": "PSH,ACK", "channel_id": "",
    "message_type": "relay"
  },
  "trace": {"round": 3, "talk": "", "correlation_id": ""}
}
```

### 前端日志接入

前端 `logEntry()` 将日志异步上报 `POST /api/logs/ingest`，标记 `source: "frontend"`、`category: "frontend"`，进入 global.jsonl。

### API 端点

| 端点 | 用途 |
|------|------|
| `GET /api/logs` | 查询日志（支持 agent_id/level/event/keyword 过滤） |
| `GET /api/logs/stats` | 日志统计 |
| `GET /api/logs/agent/{id}` | 单个 Agent 时间线 |
| `GET /api/logs/messages` | 通信报文 |
| `POST /api/logs/agent` | Agent 决策/执行日志（向后兼容） |
| `POST /api/logs/ingest` | 通用日志接入（前端/外部） |
| `GET /api/logs/export?fmt=jsonl\|json\|csv` | 导出 |
| `GET /api/logs/export/file?fmt=jsonl` | 导出到文件 |
| `GET /api/logs/files` | 历史日志文件列表 |
| `GET /api/logs/download/{path}` | 下载日志文件 |
| WebSocket `agent_log` | 实时推送（兼容旧格式缓冲区） |

### 存储与导出

- 结构化日志: JSON Lines 持久化 (`agent_network/logger.py`，v2 统一 schema)
- 报文记录: TCP/IP 包级别 — `PacketRecorder`（`event_bus.py`，**内存存储，不写入 JSONL 文件**）
- 支持导出: JSONL / JSON / CSV
- WebSocket 实时推送
