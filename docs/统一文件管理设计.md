# 统一文件管理设计

> 状态：已实现。本文定义剧本、Skill 源文件、日志、PCAP、capture manifest、experiment manifest、质量结果、分析结果和归档文件共用的文件基础设施。业务语义仍由各领域模块负责。

## 1. 设计结论

平台使用 `agent_network.file_management.FileManager` 作为受管业务文件的唯一基础设施入口，统一提供：

- 安全路径解析和稳定 `resource_id`；
- 文件、目录和外部生成文件的注册；
- 原子写入、追加写入和流式写入；
- 文本、二进制、JSON、流式及目录子文件读取；
- ZIP 压缩、虚拟条目写入和受限安全解压；
- 上传临时目录到正式目录的原子提升；
- 下载前存在性、可见性、状态和 SHA-256 校验；
- 可见性、生命周期、大小、树哈希和物理删除；
- 控制面与 Agent 容器共享注册表时的跨进程文件锁。

统一管理不要求所有文件进入同一目录。物理存储仍按 `scenes`、`logs`、`pcap`、`archives` 和 `temp` 分根管理。

## 2. 模块边界

| 模块 | 保留职责 | 委托给 FileManager 的职责 |
|---|---|---|
| `SceneStorage` | 剧本结构、Agent、Skill、Tool、拓扑校验和运行占用规则 | 发现、读取、上传、解压、提升、压缩、下载、可见性、删除 |
| `mcp_server` Skill source | Skill allowlist 和单文件/包语义 | 场景目录子文件定位、枚举和读取 |
| `LogManager` | schema、规范化、内存索引、查询和统计 | JSONL 创建、追加、列表、下载、可见性、导出和删除 |
| 抓包模块 | `tcpdump` 进程、过滤器、限额和健康状态 | 受管 PCAP 路径、资源注册、刷新、哈希和 manifest 写入 |
| `capture_management.packet_store` | PCAP 协议解析、统计和分析 | 枚举受管 PCAP、获取内部只读定位符 |
| `experiment_manifest` | 来源、完成状态和质量审计语义 | manifest、quality、analysis、sample、bundle 的读写和归档 |
| API 层 | 请求校验、业务权限和 HTTP 映射 | 只消费 `DownloadDescriptor`，不拼接或返回物理路径 |

`tcpdump` 必须把数据写入文件，因此它是允许的外部生产者；目标路径必须由 `FileManager` 的受管 `pcap` 根生成，停止后必须刷新并登记完整性元数据。

## 3. 文件资源模型

`FileResource` 包含：

| 字段 | 语义 |
|---|---|
| `resource_id` | 稳定资源标识，不是物理路径 |
| `owner_type` / `owner_id` | 所属领域和业务实例 |
| `resource_type` | `scene_directory`、`application_log`、`pcap`、`archive` 等 |
| `root_name` / `relative_path` | 仅供基础设施使用的受管定位信息 |
| `logical_name` / `media_type` | 展示、下载和归档名称 |
| `visible` | 默认列表和普通下载是否可见 |
| `state` | `ready`、`failed`、`deleted` |
| `size_bytes` / `sha256` | 完整性元数据；目录使用树清单计算来源哈希 |
| `created_at` / `updated_at` | 生命周期时间 |
| `error_message` | 最近基础设施错误 |

隐藏与删除严格分离：`hidden + ready` 可被显式内部流程读取；`deleted` 已物理删除，不能通过显示操作恢复。

## 4. 注册表与并发

统一注册表版本为 `file-registry.v1`。默认路径：

```text
data/pcap/.file_registry.json
```

选择 PCAP 根是因为该卷同时挂载到 `srv`、静态 Agent 和动态 Agent 容器。每次读写注册表前重新加载最新内容，写操作通过进程内锁和 Linux `flock` 串行化，并使用同目录临时文件加 `os.replace` 原子替换，避免多个进程覆盖彼此的资源记录。

注册表不保存文件正文，也不替代业务日志或实验 manifest。

## 5. 路径与安全规则

1. 所有受管路径必须位于已配置根目录内。
2. 相对路径拒绝绝对路径和 `..`，解析结果不得逃逸根目录。
3. 符号链接不得注册、读取或从归档中恢复。
4. 普通写入和注册表写入采用临时文件加原子替换。
5. 解压限制条目数、单文件大小和总大小，并在失败时清理临时目录。
6. 剧本上传先进入 `temp/upload`，解压到 `temp/extract`，结构校验后原子提升到 `scenes`。
7. 目录下载必须先创建受管 ZIP；PCAP 条目使用 ZIP stored。
8. 下载前重新计算 SHA-256，注册值不一致时拒绝下载。
9. API 响应不得包含 `internal_path`、宿主机路径或容器路径。

## 6. 运行数据流

### 6.1 日志

`LogManager` 在会话开始时注册会话目录，首次产生某类日志时创建稳定资源，之后通过 `append_or_create_text` 追加 JSONL。原 `.log_visibility.json` 已删除，可见性只保存在统一注册表中。

### 6.2 剧本与 Skill

`SceneStorage` 负责将已有目录注册为 `scene_directory`。列表、详情、仿真 setup、Skill 包枚举和 Skill 文件读取均从该目录资源出发。上传归档通过安全解压和原子提升进入正式根。

### 6.3 PCAP 与实验文件

Agent 容器从受管 `pcap` 根获得 `tcpdump` 输出位置，并注册 PCAP 和 capture manifest。控制面按 session 同步外部生成文件，质量审计只引用 `resource_id`。quality、analysis、packet sample 和 bundle 都是受管资源。

## 8. 验收要求

- 核心文件操作、跨实例注册表合并、路径穿越拒绝必须有单元测试；
- 日志的写入、下载、隐藏、显示和删除必须验证统一资源状态；
- 剧本的发现、上传、压缩、可见性和删除必须验证统一资源状态；
- Skill 单文件和目录包必须通过受管场景资源读取；
- PCAP、capture manifest、experiment manifest、质量审计和 bundle 必须通过资源标识关联；
- 对外数据包结果不得包含物理 PCAP 路径；
- `docker compose config` 必须证明控制面和 Agent 容器共享 PCAP 根及注册表路径。
