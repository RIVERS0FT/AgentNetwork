# ADR-030：设计驱动与可追溯变更流程

- 状态：已接受
- 日期：2026-07-20
- 适用范围：仓库内全部受版本控制的代码、配置、脚本、测试、部署和文档变更，以及影响复现结果的本地操作配置变更

## 背景

仅在代码完成后补充零散说明，无法可靠回答“为什么改、改了什么、如何验证、怎样复现”。AgentNetwork 的设计、实现和实验结果都需要可追溯，因此设计记录是变更本身的一部分，不是可选的收尾工作。

## 决定

每项修改都必须在相关设计文档中留下记录。没有设计记录、实现映射或验证证据的修改视为未完成，不得合并。

设计记录必须与实现处于同一提交或同一 PR，并至少说明：

1. 变更动机与目标；
2. 影响范围和明确不变的边界；
3. 设计决定及被放弃方案（没有替代方案时明确写“无”）；
4. 实现映射，包括代码、配置、脚本、测试或部署入口；
5. 迁移、兼容、回滚和失败语义；不适用时明确写“不适用”及原因；
6. 验证命令、预期结果和实际结果。

## 记录位置

- 改变系统职责、协议、数据模型、生命周期或不可轻易回退约束：新增或修订 ADR，并同步相关专题设计文档；
- 修复、重构、配置、脚本、测试或部署变更：更新最相关的设计文档，记录实现映射和验证证据；
- 不改变既有设计语义的修改：仍需记录“设计不变”、原因、影响范围和验证结果；
- 仅修改文档：该文档本身即为变更记录，但仍应说明它修订了什么事实或规则；
- 不受 Git 跟踪但影响复现的本地操作配置：记录在相关开发或部署设计文档中，并给出可重复执行的检查命令。

不得用提交信息、聊天记录或口头约定替代设计文档。它们可以链接设计记录，但不是长期权威来源。

## 可信变更流程

1. **读取**：开始前阅读 `docs/README.md`、本 ADR、相关专题设计和当前实现；
2. **设计**：先确定权威文档与变更边界，复杂或不可逆决定先写 ADR；
3. **实现**：同步修改实现、测试和设计记录，保持一一映射；
4. **验证**：执行与风险相称的测试，保存命令及实际结果；
5. **审阅**：检查完整 diff，确认没有范围外修改、旧引用或未记录变更；
6. **交付**：设计、实现、测试和验证证据齐全后才允许提交、推送或合并。

提交前运行零依赖门槛检查：

```bash
# 检查本地暂存、未暂存和未跟踪变更
python scripts/check_design_traceability.py

# 检查 PR 或已提交分支相对基线的完整变更
python scripts/check_design_traceability.py --base <base-ref>
```

检查器要求变更集中至少存在一个新增或更新的 `docs/*.md` 文件。它只验证设计记录存在，记录质量仍由本 ADR 的必填内容和人工审阅保证。

## 合并门槛

审阅者必须能够仅依据仓库内容建立以下证据链：

```text
需求或问题 -> 设计记录 -> 实现位置 -> 测试或检查 -> 验证结果 -> 提交/PR
```

任一环缺失时，变更不得标记为完成。若验证受环境限制无法执行，设计记录必须写明阻塞原因、未验证风险和后续责任人，不得把“未执行”描述为“已通过”。

## 2026-07-20 Git 上游修正记录

- 问题：本地 `github` 分支错误跟踪远端 `github/heads/github`，导致默认 pull/push 目标不是主线；
- 决定：将本地 `github` 分支的 upstream 统一设置为 `github/main`；
- 执行命令：`git branch --set-upstream-to=github/main github`；
- 影响边界：只修改本地 `.git/config`，不改变业务代码、远端引用或提交历史；
- 回滚方式：如需恢复其他上游，重新执行 `git branch --set-upstream-to=<remote>/<branch> github`；
- 实际验证：`branch.github.remote=github`、`branch.github.merge=refs/heads/main`，`git status --short --branch` 显示 `github...github/main [ahead 2]`。
