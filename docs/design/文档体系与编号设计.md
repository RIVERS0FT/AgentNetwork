# 文档体系与编号设计

## 目标

建立单一、可检查的文档权威结构，使需求、架构决策、详细设计、操作指南和实施记录各自承担明确职责，并能从变更原因追踪到实现与验证证据。

## 分域模型

| 目录 | 职责 | 编号规则 |
|---|---|---|
| `docs/requirements/` | 产品与系统需求基线、SR-ID | 按需求文档内的 SR 体系维护 |
| `docs/adr/` | 不可轻易回退的架构决策、状态和替代关系 | 永久三位编号 `ADR-NNN` |
| `docs/design/` | 当前系统结构、领域模型、接口和运行流程 | 使用稳定主题名，不占用 ADR 编号 |
| `docs/guides/` | 开发、部署和操作步骤 | 使用任务主题名 |
| `docs/records/` | 一次性迁移、实施和验证记录 | 使用可识别的实施主题名 |

`docs/README.md` 是分类导航入口，不替代任何权威正文。`docs/adr/README.md` 是 ADR 编号、状态和替代关系的唯一索引。

## ADR 编号与生命周期

1. 每个 ADR 编号永久对应一个独立文件，文件名格式为 `ADR-NNN-主题.md`。
2. 编号只递增、不复用；当前序列为 `ADR-001`～`ADR-032`，下一编号为 `ADR-033`。
3. 决策被修订或替代时保留原文件，并在原 ADR、新 ADR 和索引中同步记录关系。
4. 详细设计不得嵌入以 `## ADR-NNN` 开头的第二份 ADR 正文，避免形成双重权威。
5. ADR 标题中的编号必须与文件名一致，索引必须覆盖全部 ADR 文件。

## 变更流程

```text
需求或问题
  -> ADR（需要形成不可轻易回退的决定时）
  -> 详细设计
  -> 代码、配置、脚本或文档实施
  -> 测试与实际验证
  -> 实施记录或 ADR 中的实施映射
```

所有变更遵守 ADR-030。检查器只接受 `adr/`、`design/`、`requirements/` 或 `records/` 中新增或更新的 Markdown 作为权威设计记录；仅修改索引或指南不能满足追踪门槛。

## 失败与回滚语义

- 编号不连续、重复、标题不匹配或索引缺失时，文档结构测试失败；
- 本地 Markdown 链接无法解析时，文档结构测试失败；
- 变更集中没有权威设计记录时，追踪检查器失败；
- 回滚必须整体恢复目录、链接、测试路径和 ADR 状态，禁止同时保留新旧权威正文。
