# 日志与实验数据设计

## 1. 总体定义

AgentNetwork 持久化三类 JSONL：

| 日志 | 当前版本 | 定义 |
|---|---|---|
| `application.jsonl` | `application.v12` | Agent 行为、后端原生能力授权与传统应用层日志的统一事件流 |
| `network.jsonl` | `network.v4` | 由真实 PCAP 产生的原始网络层与传输层 packet 记录 |
| `system.jsonl` | `system.v3` | 编排、抓包、网络配置和内部故障排查日志 |

这三个文件边界是设计合同。不得重新引入 `layer`、`category` 等旧兼容字段来混合分类，也不得根据字段猜测并接受旧格式。

## 2. Application 日志

### 2.1 顶层身份

每条 application 记录必须包含：

```json
{
  "timestamp": "...",
  "event": "...",
  "trace_id": "...",
  "agent_id": "logical-agent-id"
}
```

`agent_id` 必须是逻辑 Agent ID，不是容器槽位名。`trace_id` 用于关联一次实验会话中的任务、消息、工具和网络证据。

### 2.2 事件族

当前允许的事件包括：

- Agent 生命周期：`agent_run_started`、`agent_run_completed`、`agent_run_failed`
- 消息：`agent_message`、`agent_message_received`
- 行为：`reasoning`、`acting`
- Skill/Tool：`skill_use`、`tool_call`、`tool_call_requested`、`tool_result`、`tool_result_received`
- 状态与策略：`state_change`、`policy_check`、`subagent_lifecycle`
- 错误：`application_error`
- 模型：`llm_api_call`、`llm_runtime_completed`

不同事件通过 schema 限制可用对象字段：`target`、`task`、`conversation`、`action`、`content`、`skill`、`tool`、`state_change`、`result`、`metrics`、`payload`。未知顶层字段不得持久化。

### 2.3 语义

- `reasoning` 和 `acting` 是统一的 Agent 行为事件。
- `agent_message` 表示发送，`agent_message_received` 表示目标 inbox 接收。
- Tool 请求与结果应使用同一个后端提供的 tool call ID（如可用）。
- 原生 Tool 在执行前同时产生 `policy_check` 和 `tool_call_requested`；拒绝请求以 denied 作为终态，允许请求必须产生 `tool_result_received`。
- `subagent_lifecycle` 使用顶层 `agent_id` 表示父 Agent，`target.agent_id` 表示子 Agent；payload 保存 backend、父 ID、子 session、run ID、模型与策略哈希。
- 原生 Tool 输入输出保存 SHA-256 和脱敏预览；文件写入额外保存 `file_before/file_after` 哈希。
- `llm_api_call` 用于 API 请求级 token 使用；`llm_runtime_completed` 用于后端运行时级时延、成本或 session 信息。
- `application.jsonl` 可以包含应用协议、HTTP/Tool/LLM 等应用层事实，但不能包含伪造的数据包。
- 目标事件驱动设计使用 `event_id` 或业务对象内的关联标识表达调度事件关系，不使用固定执行批次编号或 tick 作为仿真主时间轴。

## 3. Network 日志

每条 `network.v4` 记录表示一个真实 packet，核心结构为：

```json
{
  "timestamp": "...",
  "log_id": "net_...",
  "context": { "...": "capture attribution" },
  "network": { "...": "network and transport fields" },
  "raw": { "...": "original packet bytes and integrity" }
}
```

### 3.1 `context`

必须包含：

- `trace_id`
- `capture_id`
- `packet_index`
- `observer_agent_id`
- `runtime_container`
- `interface`
- `captured_length`
- `original_length`
- `truncated`

### 3.2 `network`

保存从原始 packet 解码得到的网络层和传输层字段，例如 IP、端口、协议、TCP flags、方向和流信息。字段可以随解析能力扩展，但来源必须是同一 packet。

### 3.3 `raw`

必须包含：

- `format`
- `encoding`
- `data`
- `byte_length`
- `packet_count`
- `sha256`

原始数据与哈希用于证明记录不是由应用事件拼接而来。

## 4. System 日志

`system.jsonl` 面向平台调试，顶层包含 `timestamp`、`level`、`source`、`debug`，可附带 `action`、`payload`、`result`、`metrics`。

适合记录：

- session start/stop；
- 事件入队、派发、完成、失败和取消；
- 网络 profile 配置与清理；
- 抓包启动、停止和健康检查；
- 事件调度器状态；
- 资源限制触发；
- 内部异常。

Agent 业务行为不应降级写入 system；应使用 application schema 中的对应事件。

## 5. 写入与校验

所有入口最终必须经过 `LogManager` schema 规范化：

- application 事件必须是允许的事件名；
- 必填字段缺失时拒绝写入；
- 对象字段按 schema 过滤；
- 不允许的顶层字段被拒绝；
- 文件按 session 存放；
- 内存索引用于实时查询，JSONL 是持久化结果。

`/api/logs/agent` 是简化适配入口，会转换成严格 application event；`/api/logs/ingest` 接收已结构化记录并推断类型。

## 6. 实验来源与质量

当前实验清单为 `agent-traffic-experiment.v3`，固定记录事件驱动调度、运行资源、原生能力策略与版本、子 Agent 树、事件统计和停止原因。不得继续沿用旧调度字段表达新语义。

目标 experiment manifest 至少记录：

- session、trace、scene、seed 和时间；
- 场景目录所有文件的 SHA-256；
- Agent 逻辑身份、后端、容器、IP、镜像身份；
- 脱敏后的 LLM 配置及其哈希；
- `event_driven` 调度模式；
- `duration_seconds`、`agent_timeout_seconds`、`idle_timeout_seconds`；
- CPU、内存、进程数和 Agent 并行度限制；
- 已处理、失败和取消的事件数量；
- 最终状态、停止原因、实际持续时间、网络与抓包结果；
- 资源释放和清理结果。
- 每个 Agent 的原生能力策略与 SHA-256；
- Claude Code、Claude Agent SDK、OpenClaw 和 OpenClaw SDK 精确版本；
- 原生子 Agent 的 parent、runtime kind、session、depth 和终态。

质量审计至少检查：

- manifest 存在且版本合法；
- manifest 不包含已废弃的固定执行计数字段；
- 每个预期 Agent 都有 capture manifest 和 PCAP；
- PCAP 头和 packet 存在；
- runtime 身份完整；
- SHA-256 匹配；
- 场景来源非空；
- trace 匹配的 application event 存在且覆盖每个 Agent；
- 事件统计与可观测日志一致；
- 实验状态为 `complete`。
- Claude Code/OpenClaw Agent 都具有原生策略身份；
- 允许的原生 Tool 请求都有结果终态，拒绝请求保留拒绝结果；
- 已启动的原生子 Agent 都具有 completed、failed、cancelled、terminated、timeout、killed、reset 或 deleted 终态。

## 7. 实验包

按需生成的 bundle 包含：

- PCAP 与 capture manifests；
- experiment manifest；
- application/network/system JSONL；
- `quality.json`；
- `analysis.json`；
- `packets.sample.jsonl`；
- `SHA256SUMS.json`。

PCAP 使用 ZIP stored，其他文本产物可压缩。

## 8. 设计约束

- 修改 schema 版本时必须同步修改测试、API、Dashboard 消费和本文件。
- experiment manifest 调度模型变更必须升级版本，并同步修改生产者、质量审计和 bundle 消费者。
- 原生能力策略、后端版本或审计完整性合同变化必须升级 manifest 或 application schema，并同步 Dashboard。
- 审计 Token、Hook 或 Plugin 不可用时不得以 system warning 替代 application 审计并继续执行。
- 不为兼容旧日志保留隐式字段映射；当前阶段直接迁移生产者和消费者。
- `application.jsonl` 与 `network.jsonl` 通过 trace、Agent 身份和时间关联，但语义边界保持独立。
- 删除日志字段前必须先删除所有生产者、消费者和测试中的依赖。
- 不得把 `round`、`tick`、`max_rounds`、`stalemate_rounds` 等旧调度字段重新定义为事件驱动仿真的权威时间或终止合同。
